# ADR-004: Inter-Service Communication & Saga Orchestration

## Status

Accepted

## Date

2026-06-05

## Context

The seven services must coordinate the order lifecycle (place → pay → accept → assign driver →
deliver) and one compensation path (payment captured → no driver → refund). ADR-001/ADR-002 require
an orchestrated saga with visible state and async-shaped seams, and recommend an in-code orchestrator
now with a workflow engine deferred. The stack is .NET (ADR-003).

## Decision

Use **RabbitMQ** as an asynchronous message broker for inter-service communication, with
**MassTransit** in .NET as the messaging abstraction. The **Order service hosts the saga** as a
MassTransit state machine (orchestration): it reacts to integration events (`OrderPlaced`,
`PaymentSettled`, `OrderAccepted`, `DriverAssigned`, `DriverUnavailable`, `OrderDelivered`) and emits
commands. The single compensation is a `RefundPayment` command issued when dispatch reports no driver.
Synchronous **REST** is used only for client-facing reads/commands through the API Gateway/BFF;
service-to-service flow is event-driven.

## Alternatives Considered

### Alternative 1: Choreography (events only, no central coordinator)

- **Description**: Each service reacts to events with no central saga.
- **Pros**: Loose coupling; no orchestrator component.
- **Cons**: The order state machine and compensation are smeared across services; hard to observe or
  reason about failure paths.
- **Why rejected**: The PRD's value is a *visible* saga with a clear compensation; choreography hides
  exactly that.

### Alternative 2: Synchronous-only (REST/gRPC) orchestration

- **Description**: The orchestrator calls services directly over HTTP/gRPC; no broker.
- **Pros**: Simplest infrastructure; easy to trace a single call chain.
- **Cons**: Contradicts the async-shaped payment seam (callback settlement) and complicates real-time
  status propagation; tight temporal coupling.
- **Why rejected**: Breaks the async settlement contract the PRD mandates and the tracking model.

## Consequences

### Positive

- A single, observable saga with explicit states and one compensation path.
- MassTransit abstracts the transport, so the broker (or a future Temporal/Cadence engine) is
  swappable without rewriting service logic.
- Async events make the SignalR real-time tracking (ADR-007) natural.

### Negative

- Adds a broker container and message-contract maintenance.
- Eventual consistency requires correlation IDs and idempotency discipline.

### Risks

- **Lost/duplicate messages** → mitigation: idempotency keys, the transactional outbox pattern in the
  Order/Payment services, and MassTransit retry/redelivery.
- **Saga stuck states** → mitigation: timeouts in the state machine that drive the order to a terminal
  state.

## Implementation Notes

- Define integration events/commands in a shared `Contracts` library consumed by all services.
- The Order service owns the saga instance store (PostgreSQL, see ADR-006).
