# ADR-005: Service Decomposition — Full Canonical Seven Services + Gateway/BFF

## Status

Accepted

## Date

2026-06-05

## Context

The product's primary purpose is to demonstrate a microservices architecture and act as a foundation
for a real product. The canonical industry breakdown (DoorDash/Uber Eats references) maps cleanly to
the PRD's flow. The decomposition granularity trades microservices fidelity against operational
weight for a mocked PoC.

## Decision

Decompose into **seven services behind an API Gateway/BFF**, each owning one bounded context:

1. **Search** — restaurant discovery/search.
2. **Catalog** — restaurants, menus, items.
3. **Order** — order aggregate + the saga orchestrator (ADR-004).
4. **Payment** — async-shaped payment behind a swappable seam.
5. **Dispatch** — driver matching/assignment behind a swappable seam (nearest-available mock).
6. **Tracking** — order-status projection + real-time fan-out source.
7. **Notification** — fire-and-forget notifications behind a swappable seam.

The **API Gateway/BFF** (YARP-based) is the single entry point for the Angular client, aggregating
reads and hosting the SignalR hub (ADR-007). Each service is an independently deployable .NET process
with its own datastore (ADR-006).

## Alternatives Considered

### Alternative 1: Pragmatic consolidation (~5-6 services)

- **Description**: Merge Search+Catalog and fold Tracking into Order.
- **Pros**: Fewer containers; simpler bootstrap and ops.
- **Cons**: Dilutes the microservices demonstration; couples search/catalog and tracking/order.
- **Why rejected**: The user prioritized full microservices fidelity.

### Alternative 2: Minimal (~3 services)

- **Description**: Storefront (search+catalog+order+tracking), Payment, Dispatch.
- **Pros**: Fastest to build and run.
- **Cons**: Barely a microservices architecture; collapses most boundaries.
- **Why rejected**: Defeats the stated learning/foundation goal.

## Consequences

### Positive

- Maximal microservices fidelity; clean bounded contexts and ownership.
- Swappable seams (Payment, Dispatch, Notification) are isolated in their own services.

### Negative

- Seven services + gateway + broker + datastores is a heavy stack for a PoC.

### Risks

- **Operational/bootstrap weight** → mitigation: a single `docker compose up` brings up the whole
  stack; one-command bootstrap is an explicit success metric.
- **Distributed-systems overhead for a demo** → mitigation: keep each service thin and happy-path
  focused; defer cross-cutting concerns (Non-Goals).

## Implementation Notes

- Shared cross-service code (event/command contracts, common middleware) lives in small shared
  libraries, not a shared domain model.
