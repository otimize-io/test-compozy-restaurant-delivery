# ADR-003: Technology Stack — .NET Backend Microservices + Angular Frontend

## Status

Accepted

## Date

2026-06-05

## Context

The repository is greenfield with no stack chosen. The idea phase deferred the technology stack to
the TechSpec. During PRD creation the user directed that the product use **.NET + Angular**. A
mandated platform is an organizational constraint that shapes the product and should be recorded as a
high-level technical constraint at the PRD level, even though the PRD otherwise avoids implementation
detail. Deeper design choices (service decomposition, messaging, persistence, orchestration engine,
specific .NET/Angular libraries and project structure) remain the TechSpec's responsibility.

## Decision

The product's V1 uses:

- **Backend:** **.NET** for the microservices (each bounded service — search, catalog, order,
  payment, dispatch, tracking, notification — implemented as a .NET service behind an API gateway/BFF).
- **Frontend:** **Angular** for the web client(s) delivering the consumer, restaurant, and driver
  views with the role switcher.

This is recorded as a fixed **High-Level Technical Constraint** in the PRD. The swappable-mock-seam
principle (ports/adapters for payment, maps/ETA, notifications) is implemented within the .NET
services.

## Alternatives Considered

### Alternative 1: Defer stack to the TechSpec (idea-phase default)

- **Description**: Leave the stack unspecified at PRD level, decide it during technical design.
- **Pros**: Keeps the PRD purely business-focused; maximum implementation freedom.
- **Cons**: Ignores an explicit user mandate; later artifacts would lack a constraint the user has
  already fixed.
- **Why rejected**: The user has decided the stack; recording it now keeps artifacts consistent and
  unblocks the TechSpec.

### Alternative 2: A different stack (e.g., Node/Go backend, React frontend)

- **Description**: Choose another common microservices/frontend combination.
- **Pros**: Each has its own ecosystem strengths.
- **Cons**: Contradicts the user's explicit direction; no stated reason to override it.
- **Why rejected**: The user explicitly chose .NET + Angular.

## Consequences

### Positive

- Unblocks the TechSpec with a known platform; aligns the team around one ecosystem.
- .NET's first-class support for service hosting, messaging, and DI fits the microservices +
  ports/adapters design; Angular's structure suits three role-based views in one app.

### Negative

- Locks V1 to one ecosystem; revisiting the stack later would be a significant change.

### Risks

- **Over-specifying implementation in the PRD** → mitigation: the PRD records only the stack mandate
  as a constraint; all design detail stays in the TechSpec.

## Implementation Notes

- The TechSpec will define .NET service boundaries, inter-service communication, the orchestration
  mechanism, persistence, and the Angular application structure (single app with role switcher vs.
  separate builds).
