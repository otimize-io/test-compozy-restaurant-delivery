# ADR-001: V1 Scope — Full Three-Sided Demonstrable Journey on Swappable Microservice Seams

## Status

Accepted

## Date

2026-06-05

## Context

The project is a mocked, microservices-based food-delivery MVP (iFood-style), intended as a
**proof-of-concept and foundation for a real product**. The greenfield repository imposes no stack
or conventions. The user set these anchors:

- All three marketplace sides (consumer, restaurant, driver) should be **interactive**.
- Depth is minimal happy-path; **payment happens before driver assignment**.
- An earlier-stated success criterion was clean, swappable service boundaries (mocks replaceable by
  real integrations without rewriting neighbors).

A five-advisor council (Pragmatic Engineer, Architect, Product Mind, Devil's Advocate, The Thinker)
and a product-strategist opportunity scan both recommended a **depth-first** path: one
"tracer-bullet" order across real seams with thin UIs, promoting the swap proof to a CI-enforced
invariant. Presented with that recommendation versus richer breadth, the **user explicitly chose
breadth** — prioritizing a demonstrable "small iFood" with three richly interactive sides, and
accepting less depth on the architecture proof.

This ADR records the scope decision actually taken, and preserves the council's depth-first analysis
as retained design principles and as the V2 stretch direction.

## Decision

V1 is scoped as a **full, demonstrable three-sided journey** on a microservices backbone:

- **Three interactive UIs:** consumer (search → menu → cart → pay → track), restaurant (receive →
  accept → prepare/ready), driver (receive assignment → pick up → deliver). The consumer side is the
  richest; restaurant and driver are interactive but leaner.
- **One happy path** end to end: `search → view menu → place order → pay → restaurant accepts → find
  & assign driver → deliver`, with **payment before driver assignment**.
- **Microservices** decomposed along industry-canonical boundaries: search/discovery, catalog/menu,
  order, payment (mock), dispatch/driver-matching (mock), tracking, notification (mock), behind an
  API gateway / BFF.
- **Swappable mock seams retained as a design principle (not the headline):** every external
  integration (payment, maps/ETA, notifications) sits behind a port/adapter so a mock can later be
  replaced by a real provider with minimal blast radius. This keeps the "foundation for a real
  product" objective alive cheaply, without making the swap proof the organizing spine.
- **Driver matching** uses a nearest-available mock behind the dispatch port.

The orchestrated saga with a compensation path (payment captured → no driver → refund) is **strongly
recommended** but is carried as an Open Question for the PRD, because the user de-prioritized
architecture depth in favor of breadth.

## Alternatives Considered

### Alternative 1: Depth-first tracer-bullet with thin UIs (council + strategist recommendation)

- **Description**: One order across all real seams; three sides present but thin; the swap proof
  promoted to a CI-enforced invariant (two adapters per port, shared contract tests); a single
  compensation path mandatory.
- **Pros**: Maximally proves the swappable-architecture criterion; lowest cost; the proven seam kit
  is the only part that compounds.
- **Cons**: Restaurant/driver are not demo-grade; weaker as a visible stakeholder artifact.
- **Why rejected**: The user prioritized a demonstrable three-sided product over architectural
  proof depth. Retained instead as the **V2 stretch direction**.

### Alternative 2: Saga + swap proof, no UIs

- **Description**: Strip to `order → pay → dispatch → refund` plus the swap test, exposed via
  API/CLI, with zero interfaces.
- **Pros**: Fastest, cheapest, fully concentrated on the architecture.
- **Cons**: No visible demo; under-delivers as something stakeholders can watch the journey move
  through.
- **Why rejected**: Conflicts directly with the user's "three interactive sides" requirement.

## Consequences

### Positive

- Produces a legible, demonstrable end-to-end product across all three marketplace roles — strong as
  a stakeholder/portfolio artifact and a realistic skeleton for a real product.
- Retaining ports/adapters keeps integrations swappable later at low present cost.
- Exercises all three load-bearing cross-boundary transitions (consumer places, restaurant accepts,
  dispatch assigns), so the contracts are validated through real use, not just diagrams.

### Negative

- Build budget spreads across three UIs plus the microservice backbone, so the *swappability* proof
  is demonstrated by design discipline rather than enforced by tests.
- Broader scope than a pure happy-path slice; higher risk of effort going into commodity UI/plumbing
  that does not compound.

### Risks

- **"Saga costume"** (council): a happy-path-only flow that never fires a compensation proves none
  of the hard distributed-rollback behavior → mitigation: strongly recommend including the single
  payment→no-driver→refund compensation path (tracked as an Open Question).
- **Sync/infallible mocks baking false shapes into callers** (council) → mitigation: keep at least
  the payment and notification ports async-shaped (async settlement via callback, idempotency key,
  declinable) even in V1, since this is cheap and prevents costly rewrites when real providers
  arrive.
- **Scope creep across three rich UIs** → mitigation: cap each side at its happy-path journey; defer
  all combinatorial edge cases (cancellation, out-of-stock, rejection, retries) to later versions.

## Implementation Notes

- Build order suggestion: stand up the order + payment + dispatch backbone and the consumer journey
  first; add restaurant and driver UIs against the same shared order lifecycle.
- Keep every external integration behind a port with a mock adapter; design the adapter interface so
  a real provider can be added later without changing the calling services.
- Defer the orchestration engine choice (in-code orchestrator vs Temporal/Cadence) to the TechSpec;
  keep the lifecycle defined as orchestration so the engine is swappable.
