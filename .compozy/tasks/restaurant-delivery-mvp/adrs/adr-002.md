# ADR-002: V1 Product Approach — Fulfillable-Order Layering, Full Three-Sided MVP

## Status

Accepted

## Date

2026-06-05

## Context

The PRD must commit to a build/rollout strategy and resolve scope refinements left open by the
idea phase: whether V1 includes a failure/compensation path, how the three roles access the demo,
and how polished each side's UI must be. The marketplace-MVP research recommends building supply
liquidity (restaurants) first and shipping the orchestration layer at v1, while the demo-legibility
finding is that the value lands when one order object updates all three views live. The user
explicitly requires that **driver finding & association be part of the MVP**, consistent with the
original prompt which named it a core area.

## Decision

Adopt **Approach A — "fulfillable-order layering"** as the V1 build sequence, with the entire
three-sided happy path (through delivery) plus one compensation path constituting the MVP:

- **MVP scope (Phase 1):** the complete happy path across all three sides —
  `search → menu → place order → pay → restaurant accepts → find & assign driver → deliver` — plus
  exactly **one compensation path** (payment captured → no driver available → refund → order in a
  terminal cancelled/refunded state). **Driver dispatch, assignment, and the driver delivery flow
  are in the MVP, not deferred.**
- **Internal build order:** order state machine + restaurant acceptance first (a fulfillable order),
  then the consumer ordering/tracking journey, then driver dispatch/assignment/delivery and the
  compensation path. Each milestone demonstrates a complete, fulfillable order.
- **Access model:** no authentication — pre-seeded identities with a **role switcher** between
  consumer / restaurant / driver views, optimized for the "one order, three live views" demo.
- **UI fidelity:** the **consumer side is rich** (search, menu, cart, checkout, and a 5-stage live
  tracking bar); **restaurant and driver are functional-clean** (order list + action buttons that
  advance the shared order state).
- **Architecture principle retained:** external integrations (payment, maps/ETA, notifications) stay
  behind swappable ports with mock adapters, keeping the "foundation for a real product" promise.

## Alternatives Considered

### Alternative 1: Approach B — Consumer-experience first

- **Description**: Build the polished consumer journey end-to-end against a simulated downstream
  first; make restaurant and driver real later; wire real cross-service events and compensation last.
- **Pros**: Shows the most visually impressive screen earliest; strong for an early stakeholder demo.
- **Cons**: The architecturally interesting cross-boundary transitions (restaurant-accept,
  dispatch-assign) land last, so risk is back-loaded; intermediate phases demo a fake order.
- **Why rejected**: Defers the real multi-service behavior and the driver leg that the user requires
  in the MVP.

### Alternative 2: Approach C — Synchronized tracer-bullet first

- **Description**: Stand up all three thin sides + one happy path wired end-to-end in Phase 1, then
  enrich the consumer UI and add compensation/polish in Phase 2.
- **Pros**: Achieves the "one order, three live views" moment soonest.
- **Cons**: Maximum parallel moving parts before any single layer is solid; compensation and
  consumer polish slip to Phase 2.
- **Why rejected**: Higher coordination risk up front, and it pushes the compensation path (now an
  MVP requirement) out of the first deliverable.

## Consequences

### Positive

- Every milestone yields a demonstrable, fulfillable order; progress is legible to stakeholders.
- The driver assignment + delivery leg and the compensation path are guaranteed in the MVP.
- Research-backed sequencing (supply/restaurant first) reduces integration risk.

### Negative

- The polished consumer screen appears after the restaurant/order backbone, not first.
- A complete MVP (all three sides + compensation) is more than a single-side slice, so Phase 1 is
  larger than a minimal demo.

### Risks

- **Scope creep across three sides + compensation** → mitigation: cap each side at its happy path;
  exactly one deterministic compensation; defer all other edge cases (Non-Goals).
- **"Saga costume" if compensation is skipped under time pressure** → mitigation: compensation is an
  explicit MVP exit criterion, not optional.

## Implementation Notes

- Deeper technical design (service decomposition, messaging, orchestration engine, data) is deferred
  to the TechSpec; this ADR fixes product scope and sequence only.
- The role switcher and pre-seeded data are demo conveniences; real identity is a later-phase concern.
