# ADR-006: Polyglot Persistence — Datastore per Service

## Status

Accepted

## Date

2026-06-05

## Context

Each of the seven services (ADR-005) owns its data. The PRD positions the system as a foundation that
demonstrates microservice autonomy. The choice is how much real persistence infrastructure to run for
a mocked PoC: a single shared engine, polyglot real stores, or in-memory.

## Decision

Adopt **polyglot persistence — a datastore chosen per service's access pattern**:

| Service | Datastore | Why |
| ------- | --------- | --- |
| Order | **PostgreSQL** | Relational order aggregate, saga instance store, transactional outbox |
| Payment | **PostgreSQL** (own database) | Idempotency keys, settlement records, transactional integrity |
| Catalog | **MongoDB** | Document model fits nested menus/items |
| Search | **Elasticsearch** | Full-text restaurant discovery |
| Dispatch | **Redis** | Driver availability + geo lookup for nearest-available matching |
| Tracking | **Redis** | Fast status reads + pub/sub fan-out to the SignalR hub |
| Notification | **Stateless** | Fire-and-forget; no store (optional outbox log only) |

Relational access uses **EF Core**; other stores use their official .NET drivers. No service reads
another service's database; cross-service data flows only via events (ADR-004).

## Alternatives Considered

### Alternative 1: Database-per-service on a single PostgreSQL

- **Description**: One Postgres instance, a separate database/schema per service.
- **Pros**: One container; durable; honors logical ownership; simplest migrations.
- **Cons**: Less didactic about polyglot autonomy; one engine for all access patterns.
- **Why rejected**: The user chose polyglot to demonstrate real per-service store autonomy.

### Alternative 2: In-memory / SQLite per service

- **Description**: Each service keeps state in memory or SQLite.
- **Pros**: Minimal bootstrap; no DB containers.
- **Cons**: No durability across restarts; least realistic foundation.
- **Why rejected**: Undercuts the "foundation for a real product" objective.

## Consequences

### Positive

- Demonstrates genuine per-service store autonomy and the right tool per access pattern.
- Each store maps to a realistic real-world choice the team can keep when integrating for real.

### Negative

- Heaviest operational footprint: PostgreSQL + MongoDB + Elasticsearch + Redis alongside RabbitMQ and
  seven services.

### Risks

- **Bootstrap weight / resource use** → mitigation: all stores containerized in one `docker compose`,
  seeded with mock data, each used minimally; document resource expectations.
- **Schema/seed drift across stores** → mitigation: per-service seed scripts run at startup; a
  one-command bootstrap validates the whole set.

## Implementation Notes

- Each service owns its migrations/seed; no shared schema. Tracking can be rebuilt purely from events,
  so its Redis state is treated as a disposable projection.
