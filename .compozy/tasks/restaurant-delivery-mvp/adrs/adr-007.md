# ADR-007: Real-Time Status Updates — SignalR WebSocket Hub

## Status

Accepted

## Date

2026-06-05

## Context

The PRD requires the consumer's 5-stage tracking bar and the three role views to reflect a role's
action in the other views in near-real-time (target ~2 s), and identifies "one order, three live
views" as the demo's credibility proof. The stack is .NET + Angular (ADR-003) with event-driven
services (ADR-004).

## Decision

Use **SignalR** (WebSocket, with automatic fallback) for server→client real-time updates. The API
Gateway/BFF hosts a **SignalR hub**. The Tracking service consumes order-status integration events
and pushes status changes to the hub, which fans them out to subscribed Angular clients (grouped per
order and per role). The Angular app uses the `@microsoft/signalr` client to update the tracking bar
and role views as events arrive.

## Alternatives Considered

### Alternative 1: REST polling

- **Description**: The frontend polls a tracking endpoint every N seconds.
- **Pros**: Simplest; no persistent connections.
- **Cons**: Not truly live; trades latency against load; can exceed the ~2 s target.
- **Why rejected**: Weakens the core demo-legibility requirement.

### Alternative 2: Server-Sent Events (SSE)

- **Description**: One-way HTTP streaming from server to client.
- **Pros**: Simple, unidirectional push.
- **Cons**: Less integrated in the .NET ecosystem than SignalR; no built-in groups/reconnect tooling.
- **Why rejected**: SignalR offers groups, reconnection, and transport fallback out of the box.

## Consequences

### Positive

- True real-time propagation; meets the ≤2 s target; reinforces the synchronized three-view demo.
- Idiomatic .NET; SignalR groups map cleanly to per-order/per-role subscriptions.

### Negative

- Adds WebSocket connection lifecycle management on client and server.

### Risks

- **Connection drops / missed events** → mitigation: SignalR auto-reconnect plus a one-shot REST
  status fetch on (re)connect to resync the current state.

## Implementation Notes

- Status events originate from the Order saga (ADR-004) and are projected by Tracking (ADR-006, Redis)
  before fan-out, so the hub never depends on a single service's availability for history.
